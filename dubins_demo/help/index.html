Dubins Path Demonstrator — Help

This application finds the shortest curvature-constrained path between two oriented positions in the plane — a start (x0,y0,θ0) and a goal (x1,y1,θ1) — for a vehicle that always moves forward and cannot turn tighter than a fixed minimum radius. It draws all six candidate path types, highlights the shortest, and shows the geometry and the math behind them.

1. Using the app

Entering configurations

The input panel has two groups, Start (drawn green) and Goal (drawn red). Each group has three entry fields:

  • X and Y — position in metres.
  • Heading — the direction the vehicle points at that configuration.

Type a value and press Enter or click away (focus out) to commit it. Non-numeric text turns the field red and posts a message in the status bar; the stored configuration is left unchanged until you enter a valid number. Typed input, dragging on the plot, and the radius spinbox are all bound to a single model, so a change in one place is reflected everywhere.

Heading convention & units

Two pairs of radio buttons control only how headings are displayed and parsed — the stored value is always canonical (radians, math convention), so toggling never drifts the underlying scenario.

ToggleOptionMeaning
Convention Angle (math) 0° = East (+X axis), counter-clockwise positive.
Azimuth (compass) 0° = North (+Y axis), clockwise positive.
Unit Degrees Headings shown and typed in degrees.
Radians Headings shown and typed in radians.

The two conventions are related by azimuth= (90∘−angle) mod360∘ , and this relation is an involution, so switching back and forth is loss-free.

Dragging & keyboard nudges

Each configuration is drawn as an arrow. On the plot you can:

  • Move — drag the base of the arrow (its tail) to reposition the configuration. Grab zones are hit-tested in pixel space (within 12 px), so the feel stays constant at any zoom level.
  • Rotate — drag the arrow head; the heading is set to the angle from the base to the cursor.

Clicking an arrow also makes it the keyboard-nudge target:

  • ← ↑ → ↓ — move the selected configuration by ±0.1 m.
  • Shift + arrow — rotate the heading by ±1°.

Every drag or nudge re-solves all six paths and redraws live.

Turn radius

The radius spinbox (starts at 0.1 m) sets the minimum turn radius r shared by every arc. Values below 0.1 are clamped. A smaller radius makes tight turns possible (paths get shorter and CCC types become feasible); a larger radius forces wide sweeping arcs. The spinbox updates the paths as you click or type.

Details table

The details panel lists every path type:

  • Columns: path type, total length in metres, and the per-segment breakdown, e.g. L 2.31 → S 5.10 → L 0.87.
  • Sortable: click a column header to sort (default: length ascending).
  • Click to highlight: clicking a feasible row highlights that path on the plot (bold, full opacity) and dims the others. The shortest feasible path is highlighted by default; your choice is kept across re-solves while it stays feasible, otherwise it falls back to the shortest.
  • Infeasible rows are grayed out and sorted to the bottom, with the reason shown in place of the segments (e.g. “turning-circle centers are more than 4r apart”).

Turning circles & animation

  • Turning-circle overlay (toggle) draws the four circles of radius r — the left and right turning circles at the start and at the goal. These are the circles every arc segment rides along, so the overlay makes the tangent constructions in section 3 visible.
  • Animation — the play/pause button runs a marker along the highlighted path at a speed you set in metres per second. It stops and resets whenever the scenario changes.

Save, load & export

  • File → Save / Load stores the scenario (start, goal, radius policy, and display preferences) as JSON. The heading is written canonically in radians so a save/load round-trip restores the scenario exactly.
  • File → Export CSV writes the highlighted path's sampled waypoints as rows of x,y,theta_rad at a step you choose (default 0.05 m). It is disabled when no path is feasible.
  • Export PNG is available through the standard matplotlib navigation toolbar (which also provides pan and zoom).

2. The Dubins problem

Problem statement

Given a start configuration (x0,y0,θ0) and a goal configuration (x1,y1,θ1), find the shortest smooth planar curve joining them whose curvature never exceeds 1/r, where r is the minimum turn radius. The curve must leave the start pointing along θ0 and arrive at the goal pointing along θ1.

Physically this models a vehicle — a car, an aircraft in level flight, a boat — moving at constant speed with a bounded steering angle: the classic “forward-only unicycle” or Dubins car.

Assumptions

  • Constant speed. The vehicle moves forward at a fixed speed; path length is therefore proportional to travel time.
  • Bounded curvature. Steering is limited, so the tightest arc the vehicle can trace has radius r (curvature ≤1/r). Optimal paths always turn at exactly this tightest radius or go straight.
  • Forward only. No reversing. (Allowing reverse gives the richer Reeds–Shepp family, which is out of scope here.)

Dubins' theorem

In 1957 Lester Dubins proved that the shortest such path is always a concatenation of at most three segments, each either a circular arc at the minimum radius (turning Left or Right) or a Straight line, and that only two families of three-letter “words” can be optimal:

  • CSC — an arc, then a straight tangent, then an arc: LSL, RSR, LSR, RSL.
  • CCC — three arcs, where the middle arc is longer than a half-circle: RLR, LRL.

That is six candidate words in total. The optimal Dubins path is simply the shortest feasible one among them — which is exactly what this app computes and draws. A CCC word can only ever win when the start and goal are close together relative to r; when they are far apart the straight middle segment of a CSC word dominates.

3. The six words

The canonical frame

All six solvers share one simplification. Translate the start to the origin, rotate so the goal lies on the positive x-axis, and scale distances by 1/r so the turn radius becomes 1. The problem reduces to (0,0,α)→(d,0,β), with three scalars. Writing θ0,1 for the two headings and letting the vector from start to goal make angle θv=atan2(y1−y0,x1−x0):

d= (x1−x0)2+(y1−y0)2 r , α=(θ0−θv)mod2π , β=(θ1−θv)mod2π.

Each solver returns three normalized segment parameters (t,p,q) in canonical units: for an arc it is the swept angle, for a straight segment it is the length. Scaling back by r gives the real lengths, and the total path length is L=r(t+p+q). The formulas below follow Shkel & Lumelsky (2001) and match the code in dubins_demo/core/dubins.py.

CSC words: outer and inner tangents

A CSC path leaves the start on a turning circle, runs straight along a common tangent to a turning circle at the goal, then arcs onto the goal heading. Which tangent is used distinguishes the four words:

  • LSL and RSR use an outer (external) tangent — both arcs turn the same way, so the two circles sit on the same side of the straight run. An outer tangent always exists, so LSL and RSR are feasible for every scenario (though not always shortest).
  • LSR and RSL use an inner (internal, crossing) tangent — the arcs turn opposite ways and the straight run crosses between the circles. An inner tangent exists only when the two turning circles do not overlap. In canonical units that condition is d2≥4 (before the heading offset); when it fails the square root below is imaginary and the word is reported infeasible.

LSL (both arcs left, outer tangent):

p= 2+d2 −2cos(α−β) +2d(sinα−sinβ) φ= atan2(cosβ−cosα,d+sinα−sinβ) t=(−α+φ)mod2π , q=(β−φ)mod2π

RSR is the mirror image: swap the roles of α and β inside the sine terms and flip the arc signs. For the inner-tangent words LSR and RSL the straight length becomes p=−2+d2+2cos(α−β)±2d(sinα+sinβ), whose radicand going negative is exactly the “circles overlap” infeasibility.

CCC words: the third-circle construction

A CCC path replaces the straight segment with a third arc. The outer two turning circles (at the start and goal) are bridged by a third circle of the same radius that is tangent to both; the vehicle rides its far side, which is why the middle arc is always greater than π (a reflex turn). Because the bridging circle must touch both outer circles, their centres can be at most four radii apart. In canonical units the existence condition is

d<4 (equivalently d<4r in real units),

together with the requirement that the cosine argument below lies in [−1,1]. When the centres are farther than 4r apart the app reports RLR and LRL as infeasible.

RLR (right, left, right):

p=arccos⁡ ( 6−d2+2cos(α−β)+2d(sinα−sinβ) 8 ) t= ( α − atan2(cosα−cosβ,d−sinα+sinβ) + p2 ) mod2π q=(α−β−t+p)mod2π

LRL mirrors this, negating the α-terms and swapping the sign of the 2d(sinα−sinβ) contribution. The middle parameter p is taken as the reflex value (the code uses the negative branch of arccos then normalizes into [0,2π)), capturing the “middle arc > π” requirement of Dubins' theorem.

4. Worked example

Take a concrete scenario:

start=(0,0,0∘) , goal=(10,5,90∘) , r=2 m.

Step 1 — canonical transform. The vector from start to goal is (10,5), so

θv=atan2(5,10)=0.4636rad (sinθv=1/5,cosθv=2/5) d=102+522=1252=5.5902 α=(0−0.4636)mod2π=5.8195 β=π2−0.4636=1.1071

Note that sinα=−1/5, cosα=2/5, sinβ=2/5, cosβ=1/5, and α−β=−π/2, so cos(α−β)=0.

Step 2 — solve one word (LSL). Since d=5.59>4 the two CCC words are already infeasible, so the shortest path is a CSC word. Plugging into the LSL formulas:

p2=2+31.25−0+2(5.5902)(−3/5)=18.25 ⇒ p=4.2720 φ=atan2(−1/5,5.5902−3/5)=atan2(−0.4472,4.2486)=−0.1049 t=(−5.8195+(−0.1049))mod2π=0.3588 q=(1.1071−(−0.1049))mod2π=1.2120

Step 3 — scale back by r=2. The canonical parameters (t,p,q)=(0.3588,4.2720,1.2120) become real lengths:

L arc: 2×0.3588=0.7175 m , S: 2×4.2720=8.5440 m , L arc: 2×1.2120=2.4241 m L=0.7175+8.5440+2.4241=11.686 m.

The remaining CSC words come out longer (RSR ≈ 35.88 m, LSR ≈ 23.43 m, RSL ≈ 24.22 m), so LSL at 11.686 m is the Dubins path for this scenario — which is exactly the row the app highlights. Enter these numbers in the app to see them reproduced in the details table.
